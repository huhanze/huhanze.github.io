Vapor服务器之Async

2018-08-07
  • Swift
  • 服务端

1. Async概述

你可能会注意到在Vapor的一些APIs中返回了通用的 Future type。如果这是你第一次听说futures，可能你会有点小疑惑，不过不用担心，vapor使它们使用起来很容易。

Promises和 futures有关联，但他们有着明显类型，Promises是用来创建futures。在大多数时候，你将会一直使用futures工作，接受Vapor‘s API返回的值,甚至，你不需要担心promises的创建。

type description mutability methods
Future Reference to an object that may not be available yet. read-only .map(to:_:) .flatMap(to:_:) do(_:) catch(_:)
Promise A promise to provide some object asynchronously. read/write succeed(_:) fail(_:)

Futures 是可供选择的，建立在回调上的异步APIs，而且可以用链式、转化的方式来实现简单闭包无法做到的事情，十分强大。

2. 转化(Transforming)

正如Swift中的optionals，futures是可以被映射的(mapped and flat-mapped)。如下是futures最常用的操作：

method signature description
map to: U.Type, _: (T) -> U 将一个future值映射为不同的value
flatMap to: U.Type, _: (T) -> Future<U> 将一个future value映射为另一个future value
transform to: U 将future value映射为可用的value

3. Map

.map(to:_:)允许你将一个future’s value转化为其他的值，由于这个future’s value可能还没获取到(可能是异步任务执行的结果)，我们必须提供一个闭包来接受这个值。

4. Flat Map

.flatMap(to:_:)可以将一个future’s value 转化为其它的future value。以flat map命名是要求你避免创建嵌套的futures(如：Future<Future<T>>)

如果我们将上边的例子用.map(to:_:)替换，那么产生的结果就是Future<Future<Response>>。

5. Transform

.transform(_:)允许你修改一个future’s value，忽略现有的值。这对于转换Future的结果特别有用，很多时候，future的实际值并不是很重要。

Future<Void>有时候被称作为一个信号，它唯一的目的就是在一些异步操作中，通知你完成或者失败的情况。

尽管我们提供了一个现有的值进行转换了，但它仍是一个转换，知道之前所有的futures都完成了(或失败了)，这个future才会完成。

7. Future

下面来看看Future一些其它不常用的方法。

7.1 Do / Catch

和Swift里的do / catch语法相似，future有do和catch方法来等待future执行的结果。

提示：

.do和.catch同时工作，如果你忘记了调用.catch，编译器会提醒你一个未使用的结果，别忘记错误的处理。

7.2 Always

你可以使用.always方法来添加一个回调，无论future是否成功或者失败，它都会被执行。

提示：

一个future可以添加任意数量的回调。

7.3 Wait

.wait()方法同步地等待future的完成，当它失败的时候，会抛出异常。

警告：

不要在路由闭包或控制器中调用此方法。

8. Promise

大多数情况下，你将会通过调用Vapor’s APIs来转化futures返回的值，但有些时候，可能需要你自己手动创建一个promise。

创建一个promise，你需要进入到一个EventLoop(事件循环)。Vapor的所有容器都有一个eventLoop属性供你使用。

注意：

一个promise只能够被完成一次，而且后边所有的completions都会被忽略。

8.1 Thread Safety

promise是线程安全的，任何线程中promise都会被完成，这就是为什么promise在被初始化的时候需要一个event-loop。promise确保完成操作后回到它的event-loop执行。

9. Event Loop

应用程序启动时，通常会为运行在CPU中的每个内核创建创建一个even loop,每一个even loop恰好有一个线程。如果你熟悉Node.js中的even loop,那么Vapor中的even loop和其类似。唯一的区别是Vapor可以在一个进程中运行多个even loop,这是因为Swift支持多线程。

每次客户端连接到服务器时，它将会被分配一个even loop。从那时起，服务端和客户端的所有通信都会被放到同一个even loop中(通过关联even loop的线程)。

even loop负责跟踪每个连接的客户端的状态。如果客户端有等待读取的请求，even loop引发阅读通知让数据可以被读取。一旦整个请求被读取，任何等待请求数据的futures都将会完成。

10. Worker

能够访问even loop的相关操作被称作为worker,Vapor中的每个容器都是一个worker

你在Vapor中使用的最常见的容器如下：

  • Application
  • Request
  • Response

你可以通过这些容器的.evenLoop属性来访问even-loop。

Vapor中的很多方法都需要当前的worker进行传递,通常会被标记为 on: Worker。如果你在路由闭包或控制器中，传递当前的请求或相应。如果在启动应用程序时需要worker,使用Application。

11. Blocking

有一条绝对关键的规则，如下：

特别注意：

不要在even loop中直接阻塞调用。

来看一个阻塞的例子 libc.sleep(_:)

sleep(_:)函数可以阻塞当前线程一定时间(参数以秒为单位)。在一个even loop中，如果你直接阻塞正在进行的任务，那么在这个阻塞工作期间，even loop无法响应任何其他被分配的客户端。换句话说，如果你在even loop中调用了sleep(5),所有其它被连接到even loop的客户端将至少延时5秒。

所以确保在后台运行阻塞工作，当使用非阻塞的方式时，使用promise通知even loop。

并非所有的阻塞调用如sleep(_:)那么明显，如果你怀疑你使用的调用可能会发生阻塞，那么你需要查看方法的内部或者寻求他人帮助，如果函数正在执行磁盘操作或网络IO操作并使用同步API(没有回调或期futures)，那么很可能会阻塞。

提示：

如果阻塞工作是你应用程序的核心部分，您应该考虑使BlockingIOThreadPool来控制创建来执行阻塞工作的线程数量,这将帮助您避免在执行阻塞工作时占用CPU时间的even loop。
